# Задание 7.3: profiles и override

## Цель

Освоить условные сервисы через `profiles` и механизм автоматического переопределения конфигурации через `docker-compose.override.yml`. Понять правила слияния Compose-файлов.

## Требования

1. Создайте компонент с тремя табами: "profiles", "override", "Правила слияния"
2. Таб "profiles" -- полный YAML с 7 сервисами:
   - `api` и `db` без profiles (основные, запускаются всегда)
   - `adminer` и `mailhog` с профилем `debug`
   - `test-runner` с профилем `test`
   - `prometheus` и `grafana` с профилем `monitoring`
3. Под YAML покажите 5 команд запуска с указанием, какие сервисы запустятся в каждом случае
4. Таб "override" -- три переключаемых вида:
   - Базовый `docker-compose.yml` (production, коммитится в Git)
   - `docker-compose.override.yml` (dev, в .gitignore)
   - Результат слияния с комментариями (что откуда взялось)
5. Под каждым видом -- пояснение роли файла
6. Таб "Правила слияния" -- таблица: скаляры (переопределяются), маппинги (мержатся), списки (конкатенируются), command/entrypoint (переопределяются)
7. Предупреждение о дублировании портов в override и пример запуска для production с `-f`

## Подсказки

- Массив команд: `{ cmd, result, desc }` для демонстрации profiles
- Три строки для override: base, dev, result
- `useState<'base' | 'dev' | 'result'>` для переключения в override

## Чеклист

- [ ] Profiles: 7 сервисов с 3 профилями (debug, test, monitoring)
- [ ] 5 команд запуска с результатами
- [ ] Override: base, dev, result -- переключаются
- [ ] Комментарии в результате слияния (что откуда)
- [ ] Таблица правил слияния (4 типа)
- [ ] Предупреждение о дублировании портов
- [ ] Пример production-запуска с -f

## Как проверить себя

1. В profiles -- каждая команда показывает правильный набор сервисов
2. В override -- результат слияния корректно объединяет base и dev
3. Правила слияния покрывают все 4 типа полей
4. Понятно, почему override НЕ используется при -f
