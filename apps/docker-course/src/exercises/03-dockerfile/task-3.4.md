# Задание 3.4: Multi-stage builds

## Цель

Освоить многоэтапную сборку Docker-образов для создания компактных production-образов, научиться разделять этапы сборки и runtime.

## Требования

1. Создайте компонент с минимум 3 примерами multi-stage builds: Node.js API, React + Nginx, Go API
2. Для каждого примера покажите этапы сборки в виде карточек: имя этапа, базовый образ, назначение, содержимое, размер
3. Реализуйте визуальное сравнение размеров: single-stage vs multi-stage build с процентом уменьшения
4. Добавьте кнопку показа/скрытия полного Dockerfile для каждого примера
5. Добавьте блок с ключевыми принципами multi-stage builds

## Подсказки

- Для каждого примера определите массив этапов (stages) с полями: name, baseImage, purpose, contents, size, color
- Карточки этапов расположите горизонтально через flex
- Для сравнения размеров используйте полоски разной ширины (пропорционально размеру)
- Go-пример с `scratch` образом даёт самое впечатляющее уменьшение размера
- Dockerfile-примеры должны быть production-ready: USER, HEALTHCHECK, EXPOSE

## Чеклист

- [ ] Реализованы минимум 3 примера (Node.js, React, Go)
- [ ] Для каждого примера показаны этапы сборки в виде карточек
- [ ] Карточки содержат: имя, базовый образ, назначение, содержимое, размер
- [ ] Визуальное сравнение размеров single-stage vs multi-stage
- [ ] Показан процент уменьшения размера
- [ ] Кнопка показа/скрытия Dockerfile работает
- [ ] Dockerfile-примеры используют best practices (USER, HEALTHCHECK)
- [ ] Блок с принципами multi-stage builds присутствует

## Как проверить себя

1. Переключайте примеры -- каждый должен показывать свои этапы и размеры
2. Проверьте размеры: Go-приложение должно показать уменьшение с ~1.2 GB до ~10 MB
3. Откройте Dockerfile -- он должен содержать COPY --from=builder и быть production-ready
4. Убедитесь, что принципы multi-stage builds корректны
