# Задание 0.1: Типобезопасный HTTP-клиент

## 🎯 Цель

Создать generic-обёртку над `fetch`, которая обеспечивает полную типобезопасность: типизированные ответы, обязательное/опциональное тело запроса и discriminated union для результата.

## Требования

1. Создайте интерфейс `ApiEndpoint<TResponse, TBody>` с фантомными типами для хранения информации о типах ответа и тела запроса
2. Реализуйте фабричную функцию `endpoint()` для создания типизированных дескрипторов эндпоинтов
3. Создайте тип `ApiResult<T>` как discriminated union с вариантами `ok: true` (данные) и `ok: false` (ошибка)
4. Реализуйте функцию `typedFetch()`, которая:
   - Принимает эндпоинт и опционально тело запроса
   - Требует body только для эндпоинтов с типом тела (POST/PUT)
   - Запрещает body для эндпоинтов без тела (GET/DELETE)
   - Возвращает `Promise<ApiResult<TResponse>>`
5. Создайте API-каталог с минимум 4 эндпоинтами (GET, POST, PUT, DELETE)

## Чеклист

- [ ] `ApiEndpoint` использует фантомные типы для `TResponse` и `TBody`
- [ ] `typedFetch(api.getUsers)` возвращает `Promise<ApiResult<User[]>>`
- [ ] `typedFetch(api.createUser, body)` требует тело запроса
- [ ] `typedFetch(api.getUsers, body)` вызывает ошибку компиляции
- [ ] `typedFetch(api.createUser)` без body вызывает ошибку компиляции
- [ ] `ApiResult` обрабатывается через проверку `result.ok`

## Как проверить себя

Попробуйте вызвать `typedFetch` с неверными аргументами — TypeScript должен показать ошибку. Проверьте, что автодополнение работает для полей ответа после проверки `result.ok`.
