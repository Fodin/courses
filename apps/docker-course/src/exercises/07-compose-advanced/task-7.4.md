# Задание 7.4: Compose Watch

## Цель

Освоить Compose Watch -- механизм автоматической синхронизации файлов при разработке. Понять три действия (`sync`, `rebuild`, `sync+restart`), настроить watch для реального проекта и сравнить с bind mount.

## Требования

1. Создайте компонент с тремя табами: "Действия watch", "Полный пример", "Watch vs Bind Mount"
2. Таб "Действия watch" -- три кнопки переключения: `sync`, `rebuild`, `sync+restart`. Для каждого:
   - Описание действия
   - Когда использовать
   - YAML-пример конфигурации
   - Пояснение поведения
3. Под примером покажите команду запуска: `docker compose watch`
4. Таб "Полный пример" -- docker-compose.yml с двумя сервисами (api + frontend), каждый с несколькими правилами watch:
   - api: sync для src, rebuild для package.json, sync+restart для .env
   - frontend: sync для src, rebuild для package.json, sync+restart для vite.config.ts
5. Под YAML -- карточки с количеством правил для каждого сервиса
6. Таб "Watch vs Bind Mount" -- таблица сравнения по 5 аспектам: node_modules, производительность macOS, фильтрация файлов, автоматическая пересборка, сложность настройки
7. Под таблицей -- два блока кода: пример bind mount и пример Compose Watch бок о бок

## Подсказки

- Массив для действий: `{ action, desc, when, yaml, details }`
- Для ignore-паттернов: `'**/*.test.ts'`, `'**/__tests__/**'`
- Для sync нужен `target`, для rebuild -- не нужен

## Чеклист

- [ ] Три действия watch (sync, rebuild, sync+restart) с переключением
- [ ] Каждое действие: описание, когда использовать, YAML, пояснение
- [ ] Полный пример с 2 сервисами и 7 правилами watch
- [ ] Таблица сравнения Watch vs Bind Mount (5 строк)
- [ ] Код-примеры bind mount и watch рядом
- [ ] Команда запуска docker compose watch

## Как проверить себя

1. Переключение действий -- YAML и описание обновляются
2. Полный пример содержит оба сервиса с разными правилами
3. Сравнение объективно показывает плюсы watch и bind mount
4. sync использует target, rebuild -- нет (это важное различие)
