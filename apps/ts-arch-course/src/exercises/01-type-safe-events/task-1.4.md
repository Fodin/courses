# Задание 1.4: Основы Event Sourcing

## 🎯 Цель

Реализовать типобезопасное хранилище событий (EventStore) с replay, фильтрацией по типу и time-travel.

## Требования

1. Определите базовый интерфейс `BaseEvent` с полями id, timestamp, version
2. Создайте минимум 4 типа событий для одной доменной модели (discriminated union)
3. Реализуйте `EventStore<TEvent>` с методами:
   - `append(event)` — добавление события
   - `getAll()` — все события
   - `getByType(type)` — фильтрация с сужением типа через Extract
   - `replay(reducer, initial)` — восстановление состояния из всех событий
   - `replayUntil(reducer, initial, timestamp)` — time-travel
4. Создайте reducer для восстановления состояния из последовательности событий
5. Продемонстрируйте полный цикл: создание событий -> replay -> time-travel

## Чеклист

- [ ] Все события наследуют BaseEvent и используют discriminated union (поле type)
- [ ] `getByType('MoneyDeposited')` возвращает `MoneyDeposited[]` (не общий union)
- [ ] `replay` корректно восстанавливает состояние из последовательности событий
- [ ] `replayUntil` позволяет "путешествовать во времени"
- [ ] Reducer использует exhaustive matching для всех типов событий

## Как проверить себя

Проверьте, что `getByType` сужает тип корректно — у элементов результата должны быть доступны только поля конкретного типа события. Time-travel должен показывать промежуточное состояние.
