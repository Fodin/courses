# Задание 3.2: CMD vs ENTRYPOINT

## Цель

Понять разницу между CMD и ENTRYPOINT, научиться выбирать правильную инструкцию и форму записи (exec vs shell) для различных сценариев.

## Требования

1. Создайте интерактивный компонент с набором сценариев запуска контейнера (минимум 6)
2. Для каждого сценария покажите: Dockerfile-инструкцию, команду `docker run`, результат выполнения
3. Сценарии должны включать: CMD exec-форма, CMD shell-форма, CMD override, ENTRYPOINT exec-форма, ENTRYPOINT + аргументы, ENTRYPOINT + CMD, ENTRYPOINT + CMD override
4. Добавьте таблицу сравнения exec-формы vs shell-формы (по кнопке): PID 1 процесс, обработка SIGTERM, graceful shutdown, подстановка переменных, синтаксис
5. Добавьте секцию с паттерном entrypoint.sh (по кнопке): пример скрипта с `exec "$@"` и его использование

## Подсказки

- Для каждого сценария создайте объект с полями: name, dockerfile, runCommand, result
- Используйте тёмную тему для блоков кода (background: #1e1e1e)
- Разделите блок кода на секции: Dockerfile, Команда запуска, Результат
- Для таблицы exec vs shell подсветите exec-форму зелёным, shell-форму красным

## Чеклист

- [ ] Реализовано минимум 6 сценариев с переключением
- [ ] Каждый сценарий показывает Dockerfile, команду docker run и результат
- [ ] Таблица exec vs shell содержит минимум 4 строки
- [ ] Таблица показывается/скрывается по кнопке
- [ ] Секция с паттерном entrypoint.sh показывается/скрывается по кнопке
- [ ] Пример entrypoint.sh содержит `exec "$@"` и пояснение

## Как проверить себя

1. Переключайте сценарии -- убедитесь, что каждый показывает уникальную комбинацию CMD/ENTRYPOINT
2. Проверьте, что результат выполнения логически соответствует Dockerfile и команде запуска
3. Откройте таблицу exec vs shell -- exec-форма должна быть помечена как рекомендуемая
4. Откройте паттерн entrypoint.sh -- пример должен быть понятным и содержать пояснение
