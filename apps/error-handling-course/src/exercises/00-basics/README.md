# 🔥 Уровень 0: Основы обработки ошибок в JavaScript

## 🎯 Введение

Ошибки — неизбежная часть программирования. Файл не найден, сервер не отвечает, пользователь ввёл некорректные данные. Правильная обработка ошибок — это то, что отличает надёжное приложение от хрупкого.

В JavaScript есть встроенный механизм обработки ошибок: конструкция `try/catch/finally` и объект `Error`.

## 🔥 Конструкция try/catch

```javascript
try {
  // Код, который может вызвать ошибку
  const data = JSON.parse(jsonString)
  console.log(data)
} catch (error) {
  // Код обработки ошибки
  console.error('Ошибка парсинга:', error)
}
```

### Как это работает

1. Выполняется код в блоке `try`
2. Если ошибки нет — блок `catch` пропускается
3. Если ошибка возникает — выполнение `try` прерывается, управление переходит в `catch`
4. Параметр `error` в `catch` содержит объект ошибки

### Блок finally

Блок `finally` выполняется **всегда** — и после успешного `try`, и после `catch`:

```javascript
try {
  const file = openFile('data.txt')
  processFile(file)
} catch (error) {
  console.error('Ошибка обработки файла:', error)
} finally {
  // Выполнится в любом случае
  closeFile(file)
}
```

💡 Типичные сценарии для `finally`:
- Освобождение ресурсов (закрытие соединений)
- Скрытие индикатора загрузки
- Очистка временных данных

## 🔥 Оператор throw

Вы можете создавать свои ошибки с помощью `throw`:

```javascript
function divide(a, b) {
  if (b === 0) {
    throw new Error('Деление на ноль невозможно')
  }
  return a / b
}

try {
  const result = divide(10, 0)
} catch (error) {
  console.error(error.message) // "Деление на ноль невозможно"
}
```

📌 **Важно:** `throw` можно использовать с любым значением, но рекомендуется всегда бросать экземпляры `Error`:

```javascript
// ❌ Плохо — throw с примитивами
throw 'что-то пошло не так'
throw 404
throw { message: 'ошибка' }
```

> **Почему это ошибка:** при `throw` примитива (строки, числа) или обычного объекта теряется стек вызовов (`stack`). Без стека невозможно определить, где именно произошла ошибка, а `instanceof Error` вернёт `false`, ломая любую проверку типов.

```javascript
// ✅ Хорошо — всегда бросаем экземпляр Error
throw new Error('что-то пошло не так')
throw new TypeError('ожидалась строка')
```

## 📌 Объект Error

Объект `Error` имеет три основных свойства:

```javascript
const error = new Error('Что-то сломалось')

console.log(error.name)    // "Error"
console.log(error.message) // "Что-то сломалось"
console.log(error.stack)   // Стек вызовов (многострочная строка)
```

### Свойство name

Имя типа ошибки. Для базового `Error` — `"Error"`, для `TypeError` — `"TypeError"` и т.д.

### Свойство message

Текст сообщения об ошибке, переданный в конструктор.

### Свойство stack

Стек вызовов — показывает цепочку вызовов функций, которая привела к ошибке. Незаменимо при отладке:

```
Error: Что-то сломалось
    at processData (app.js:15:11)
    at loadUser (app.js:8:3)
    at main (app.js:2:1)
```

## 📌 Встроенные типы ошибок

JavaScript предоставляет несколько встроенных типов ошибок:

| Тип | Когда возникает |
|-----|-----------------|
| `Error` | Базовый тип ошибки |
| `TypeError` | Операция с неподходящим типом |
| `RangeError` | Число вне допустимого диапазона |
| `SyntaxError` | Синтаксическая ошибка при парсинге |
| `ReferenceError` | Обращение к несуществующей переменной |
| `URIError` | Некорректное использование URI-функций |
| `EvalError` | Ошибка, связанная с `eval()` (устарело) |

## 🎯 Проверка типа ошибки

Используйте `instanceof` для определения типа ошибки:

```javascript
try {
  JSON.parse('invalid')
} catch (error) {
  if (error instanceof SyntaxError) {
    console.log('Некорректный JSON')
  } else if (error instanceof TypeError) {
    console.log('Ошибка типа')
  } else {
    console.log('Неизвестная ошибка:', error)
  }
}
```

## ⚠️ Частые ошибки новичков

### 🐛 1. Пустой catch

```javascript
// ❌ Плохо — ошибка проглатывается
try {
  doSomething()
} catch (error) {
  // пусто
}
```

> **Почему это ошибка:** пустой `catch` полностью скрывает ошибку. Программа продолжает работать с невалидным состоянием, баги становятся «невидимыми» — вы не получаете ни логов, ни уведомлений. Отладка превращается в кошмар, потому что ошибка молча проглочена где-то выше по стеку.

```javascript
// ✅ Хорошо — логируем и/или пробрасываем
try {
  doSomething()
} catch (error) {
  console.error('Ошибка в doSomething:', error)
  // или пробросить дальше
  throw error
}
```

### 🐛 2. Слишком широкий catch

```javascript
// ❌ Плохо — ловит все ошибки одинаково
try {
  // 100 строк кода
} catch (error) {
  alert('Произошла ошибка')
}
```

> **Почему это ошибка:** слишком широкий `try` ловит ВСЕ ошибки — включая те, которые вы не ожидали (опечатки, `TypeError`, ошибки в бизнес-логике). Вы не можете адекватно обработать каждую из них одним сообщением. В итоге реальные баги маскируются за общим «Произошла ошибка».

```javascript
// ✅ Хорошо — конкретная обработка
try {
  const data = JSON.parse(input)
} catch (error) {
  if (error instanceof SyntaxError) {
    showValidationError('Некорректный формат данных')
  } else {
    throw error // Пробрасываем неожиданные ошибки
  }
}
```

### 🐛 3. Не проверять тип error

В TypeScript `catch` параметр имеет тип `unknown`:

```typescript
// ❌ Плохо — обращение к свойству unknown
try {
  something()
} catch (error) {
  // error имеет тип unknown!
  console.log(error.message) // ошибка TS
}
```

> **Почему это ошибка:** в `catch` может прилететь что угодно — не только `Error`, но и строка, число, `undefined`. Обращение к `.message` без проверки типа приведёт к ошибке компиляции в TypeScript, а в рантайме — к потенциальному `TypeError`, если кто-то сделал `throw 'строка'`.

```typescript
// ✅ Хорошо — проверяем тип перед использованием
try {
  something()
} catch (error) {
  if (error instanceof Error) {
    console.log(error.message) // безопасно
  }
}
```

## 📌 Итоги

- ✅ `try/catch/finally` — основной механизм обработки ошибок в JS
- ✅ `throw new Error('...')` — создание и выброс ошибки
- ✅ Объект `Error` содержит `name`, `message` и `stack`
- ✅ Всегда проверяйте тип ошибки через `instanceof`
- ❌ Не проглатывайте ошибки в пустом `catch`
- 📌 В TypeScript `error` в `catch` имеет тип `unknown`
